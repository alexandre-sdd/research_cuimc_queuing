\section{Purpose and Model at a Glance}

This note defines the first discrete-event appointment simulation used in this project and
clarifies the notation so it matches the underlying dynamics of the problem.
The main design goal is to keep the first simulator simple enough to run, inspect, and extend,
while using a formulation that remains valid once we add richer allocation policies,
heterogeneous patient behavior, or optimization layers.

The note aligns the simulator with the queueing models in:
\begin{itemize}[nosep]
    \item Green and Savin (2008), \emph{Reducing Delays for Medical Appointments: A Queueing Approach},
    \item Liu and Ziya (2014), \emph{Panel Size and Overbooking Decisions for Appointment-based Services under Patient No-shows},
    \item Liu (2015), \emph{Optimal Choice for Appointment Scheduling Window under Patient No-Show Behavior}.
\end{itemize}

All three papers study the same structural mechanism:
\begin{keybox}[title={Core Feedback Loop}]
\[
\text{more backlog} \Rightarrow \text{more delay} \Rightarrow \text{worse attendance}
\Rightarrow \text{lower effective capacity.}
\]
\end{keybox}

The simulation keeps that mechanism, but implements it on an explicit rolling calendar with
two patient classes, explicit cancellations, and a pluggable slot-allocation rule.

\begin{keybox}[title={Model at a Glance}]
\begin{itemize}[nosep]
    \item The simulator evolves at slot-sized decision epochs \(t=(D,s)\), where \(D\) is the day and \(s\) is the active slot within that day.
    \item The true state is the rolling calendar \(Y_t(r,m)\), which records the content of slot \(m\) on day offset \(r\).
    \item The derived day-start summary \(X_{i,r}^{D}\) counts how many class-\(i\) patients are scheduled for day \(D+r\) at the start of day \(D\).
    \item Patient behavior is driven by the delay \(\tau\) seen when the slot is offered, through \(b_i(\tau)\), \(\phi_i(\tau)\), and \(\xi_i(\tau)\).
    \item Demand is parameterized by total arrival rate \(\lambda\) and class-1 demand share \(p\), with \(\lambda_1=p\lambda\) and \(\lambda_2=(1-p)\lambda\).
\end{itemize}
\end{keybox}

The note proceeds in six steps. It first states the modeling choice and notation, then defines the
core formulation, then describes the baseline dynamics, then walks through one integrated FCFS
example, then defines the performance measures, and finally positions the simulator relative to the
literature and the current codebase.

\subsection{Notation Choice}

A naive notation might write
\[
X_{i,\tau}^{D} = \text{number of class-$i$ patients booked for day } D+\tau
\]
and then also use the same symbol \(\tau\) inside \(b_i(\tau)\) and \(\xi_i(\tau)\).
This note avoids that formulation for three reasons:
\begin{enumerate}[nosep]
    \item it overloads the delay variable, because the simulator needs both a \emph{residual delay} in the state and a \emph{booking-time delay} in the behavior functions;
    \item it treats the aggregate count \(X_{i,r}^{D}\) as if it were the full state, even though the simulator must track slot-level holes and positions explicitly;
    \item it obscures the link to the literature, where attendance depends on the congestion or delay seen when the appointment is accepted.
\end{enumerate}

The resulting modeling choice is simple: use \(r\) for residual delay in the state, use \(\tau\) for
the booking-time delay faced by the patient, and treat the slot-level calendar \(Y_t(r,m)\) as the
true state of the simulator.

\section{Core Formulation}

\subsection{Time Index}

Let time advance in slot-sized decision epochs.
Write
\[
t = (D,s),
\]
where:
\begin{itemize}[nosep]
    \item $D$ is the calendar day,
    \item $s \in \{0,1,\dots,S-1\}$ is the active slot within day $D$.
\end{itemize}

In the first version of the simulator,
\[
H = 15 \text{ days}, \qquad S = 25 \text{ slots per day},
\]
where $H$ is the booking horizon.

\subsubsection{How Time Advances}

The ordered pair $t=(D,s)$ advances lexicographically:
\[
(D,s)^+
=
\begin{cases}
(D,s+1), & \text{if } s < S-1,\\
(D+1,0), & \text{if } s = S-1.
\end{cases}
\]

So the slot index $s$ moves within a day, and the day index $D$ increases only when the
current day has exhausted all $S$ slots.

The same evolution can be visualized as a state-transition graph:

\subsection{State Representation}

The true simulator state at time $t=(D,s)$ is the rolling slot-level calendar.
The mathematical state is indexed by residual day $r$ and slot index $m$.
For visualization, however, it is often more natural to display the calendar with days as
\emph{columns} and slots as \emph{rows}. In that display, the column index corresponds to the day
ahead relative to the current day $D$: $r=0$ means ``today'', $r=1$ means ``tomorrow'', and so on.
The row index corresponds to the slot within a day: $m=0$ is the first slot, $m=1$ the second, and
so on until $m=S-1$.

Formally, for each residual day offset
\[
r \in \{0,1,\dots,H-1\}
\]
and each slot index
\[
m \in \{0,1,\dots,S-1\},
\]
define
\[
Y_t(r,m).
\]

So $Y_t(r,m)$ is the content of the cell indexed by day-ahead $r$ and slot $m$ in the rolling
calendar.
\[
Y_t(r,m)=
\begin{cases}
0, & \text{if the slot is open},\\[0.4em]
\left(i,\tau\right), & \text{if the slot is occupied by a class-}i\text{ patient who originally accepted delay }\tau.
\end{cases}
\]

The class index $i$ links that booked patient to the class object, written later as
\[
C_i = \{\lambda_i, b_i(\tau), \phi_i(\tau), \xi_i(\tau)\}.
\]
So the cell itself stores only the patient features that must remain attached to the appointment:
the class identity $i$ and the original offered delay $\tau$.
The remaining class characteristics are recovered from $C_i$ when the simulator needs them:
\begin{itemize}[nosep]
    \item $\lambda_i$ governs future arrivals from class $i$,
    \item $b_i(\tau)$ governs booking acceptance when a class-$i$ request is offered delay $\tau$,
    \item $\phi_i(\tau)$ governs daily pre-appointment cancellation for class-$i$ patients booked with delay $\tau$,
    \item $\xi_i(\tau)$ governs no-show at appointment time.
\end{itemize}

\subsubsection{Derived Day-Start Summary}

Although the slot-level calendar is the true state, it is still useful to retain an aggregate
day-start summary that resembles the literature.
At the start of day $D$, after the previous day's end-of-day cancellations and before processing
slot $0$,
define
\[
X_{i,r}^{D}
=
\sum_{m=0}^{S-1}
\mathbf{1}\{Y_{(D,0)}(r,m)\text{ is occupied by a class-}i\text{ patient}\},
\]
for
\[
i \in \{1,2\}, \qquad r \in \{0,1,\dots,H-1\}.
\]

In words, $X_{i,r}^{D}$ counts how many class-$i$ patients appear in row $r$ of the calendar at
the start of day $D$.
Equivalently,
\[
X_{i,r}^{D}
=
\text{number of class-}i\text{ patients scheduled at the start of day }D\text{ for day }D+r.
\]

This is the correct interpretation of the state-count variable.
The index $r$ is a \emph{residual delay}, not a booking-time delay.

\subsubsection{Booking-Time Delay}

Now define the behavioral delay variable separately:
\[
\tau
=
\text{delay in days between the booking day and the appointment day offered to the patient.}
\]

If a patient arrives on day $d$ and is offered a slot on day $d+\tau$, then:
\begin{itemize}[nosep]
    \item $b_i(\tau)$ is the probability that class-$i$ declines the offer,
    \item if the patient accepts, $\tau$ is stored with the booking,
    \item $\xi_i(\tau)$ is the no-show probability used at appointment time if the patient remains scheduled.
\end{itemize}

\begin{keybox}[title={Key Modeling Distinction}]
Residual delay $r$ belongs to the state.
Booking-time delay $\tau$ belongs to the behavioral rules.
\end{keybox}

\subsection{Patient Classes and Arrival Mix}

The class-specific behavior of each class $i \in \{1,2\}$ is described by
\[
B_i = \{b_i(\tau), \phi_i(\tau), \xi_i(\tau)\}.
\]

In the current two-class prototype, arrivals are parameterized at the system level by
\[
\lambda
=
\text{total expected arrival rate per slot},
\qquad
p \in [0,1]
=
\text{share of class-1 demand}.
\]

The class-specific arrival rates are then induced by
\[
\lambda_1 = p\lambda,
\qquad
\lambda_2 = (1-p)\lambda.
\]

Equivalently, the full class objects can still be written as
\[
C_i = \{\lambda_i, b_i(\tau), \phi_i(\tau), \xi_i(\tau)\},
\]
but in the first simulation experiments the primitive demand parameters are
$(\lambda,p)$ rather than $(\lambda_1,\lambda_2)$.

The interpretation is therefore:
\begin{itemize}[nosep]
    \item $\lambda$: total expected arrival rate per slot,
    \item $p$: fraction of arrivals belonging to class 1,
    \item $\lambda_i$: implied arrival rate of class-$i$ requests per slot,
    \item $b_i(\tau)$: probability the patient declines the offered slot because the offered delay is too large,
    \item $\phi_i(\tau)$: daily probability that a class-$i$ patient booked with original delay $\tau$ cancels at the end of a pre-appointment day,
    \item $\xi_i(\tau)$: no-show probability at appointment time, conditional on the patient still being scheduled.
\end{itemize}

In the current first version, the daily cancellation function $\phi_i(\tau)$ is induced by a
class-level eventual cancellation parameter $\bar{\phi}_i$ according to
\[
\phi_i(\tau)=
\begin{cases}
0, & \tau=0,\\[0.4em]
1-(1-\bar{\phi}_i)^{1/\tau}, & \tau\ge 1,
\end{cases}
\]
so that a patient who booked with delay $\tau$ and faces one cancellation trial at the end of each
pre-appointment day still has total cancellation probability $\bar{\phi}_i$ before the appointment.
Later versions may generalize this further to a directly delay-sensitive cancellation object.

\section{Dynamics and Baseline Policy}

\subsection{Slot-Level Event Order}

At each active slot $(D,s)$, the simulator executes the following within-day steps.

\begin{enumerate}[itemsep=0.35em]
    \item Generate arrivals
    \[
    A_i(D,s) \sim \mathrm{Poisson}(\lambda_i), \qquad i \in \{1,2\}.
    \]
    In the current parameterization, $\lambda_1=p\lambda$ and $\lambda_2=(1-p)\lambda$.
    \item Randomize the order of individual arrivals within the slot.
    \item For each arrival, call the allocation policy to select an open future slot.
    \item If a slot on day $D+r$ is offered, interpret
    \[
    \tau = r.
    \]
    \item The patient accepts with probability
    \[
    1 - b_i(\tau).
    \]
    \item When the active slot is served, the patient in that slot, if any, attends with probability
    \[
    1 - \xi_i(\tau),
    \]
    where $\tau$ is the original delay stored when the patient booked.
\end{enumerate}

After all $S$ slots of day $D$ have been processed, every still-scheduled future appointment
stored as $(i,\tau)$ cancels independently with probability $\phi_i(\tau)$.
Any canceled future slot is therefore reopened before day $D+1$ begins.
This timing is operationally natural:
\begin{itemize}[nosep]
    \item a booking with $\tau=1$ gets exactly one chance to cancel, namely at the end of the booking day,
    \item a booking with $\tau=2$ gets two daily cancellation opportunities before service,
    \item the total cancellation probability across those daily trials remains $\bar{\phi}_i$.
\end{itemize}

\subsection{FCFS Baseline Policy}

In the baseline version, the allocation rule is:
\begin{quote}
choose the earliest open slot strictly after the active slot.
\end{quote}

That rule is first-come first-served in calendar order. In this note it is the only policy under
study, because the goal is to isolate the basic delay-feedback mechanism before considering richer
allocation logic.

\(X_{i,r}^{D}\), and the implied booking decision in one place.

Take
\[
D=0, \qquad s=0, \qquad H=3, \qquad S=4,
\]
so we are at the start of the current operating day, just before the 08:00 slot, and the rolling

A realistic FCFS-consistent miniature appointment book looks like this:

This example is read column by column. Today and tomorrow are filled first, overflow appears on
day \(D+2\), and the open 15:00 slot on day \(D\) is a cancellation hole rather than a violation
of FCFS.

The same calendar can be written directly as a state snapshot:
\[
\begin{array}{c|cccc}
& \text{08:00} & \text{10:00} & \text{13:00} & \text{15:00} \\
\hline
r=0\text{ (today)} & (1,0) & (2,0) & (1,0) & 0 \\
r=1\text{ (tomorrow)} & (2,1) & (1,1) & (2,1) & (1,1) \\
r=2\text{ (day after)} & (2,2) & 0 & 0 & 0
\end{array}
\]

So, for example:
\begin{itemize}[nosep]
    \item \(Y_{(0,0)}(0,0)=(1,0)\) means that today's 08:00 slot is booked by a class-1 MRI patient who accepted a same-day offer;
    \item \(Y_{(0,0)}(1,3)=(1,1)\) means that tomorrow's 15:00 slot is booked by a class-1 MRI patient who accepted a one-day delay;
    \item \(Y_{(0,0)}(2,0)=(2,2)\) means that the 08:00 slot two days from now is booked by a class-2 behavioral-health patient who accepted a two-day delay.
\end{itemize}

Because the last patient belongs to class \(2\), the simulator links that slot to
\[
C_2=\{\lambda_2,b_2(\tau),\phi_2(\tau),\xi_2(\tau)\},
\]
so future class-2 arrivals are generated with rate \(\lambda_2\), daily cancellation while the
patient waits is governed by \(\phi_2(2)\), and eventual no-show is governed by \(\xi_2(2)\).

From the same calendar, the derived day-start counts are:
\[
X_{1,0}^{0}=2,\quad X_{2,0}^{0}=1,
\qquad
X_{1,1}^{0}=2,\quad X_{2,1}^{0}=2,
\qquad
X_{1,2}^{0}=0,\quad X_{2,2}^{0}=1.
\]

So \(X_{i,r}^{0}\) is just a class-by-class count of occupied entries in row \(r\). It does
\emph{not} remember which slot inside the day is occupied, which is why \(X_{i,r}^{D}\) is a
useful summary but not the full simulator state.

The same snapshot also shows why \(r\) and \(\tau\) must be separated. The patient stored as
\[
Y_{(0,0)}(2,0)=(2,2)
\]
contributes to \(X_{2,2}^{0}\) on day \(0\), then to \(X_{2,1}^{1}\) on day \(1\) if still
scheduled. The residual delay changes from \(r=2\) to \(r=1\), but the booking-time delay remains
\(\tau=2\), so the same behavioral functions \(b_2(\tau)\), \(\phi_2(\tau)\), and \(\xi_2(\tau)\)
continue to apply.

Finally, the calendar transfers directly into the booking equations. Suppose a new patient arrives
just before today's 08:00 slot. Under FCFS, the earliest open slot is today's 15:00 slot. If the
arrival belongs to class 1, then:
\begin{itemize}[nosep]
    \item the offered delay is \(\tau=0\);
    \item the patient books with probability \(1-b_1(0)\);
    \item if the patient accepts, the new slot content becomes \(Y_{(0,0)}(0,3)=(1,0)\);
    \item because \(\tau=0\), there is no remaining pre-appointment day on which cancellation can occur;
    \item if the patient reaches the appointment, attendance occurs with probability \(1-\xi_1(0)\).
\end{itemize}

So the same worked example can be read in three ways: as an appointment book, as a state
realization \((Y,X)\), and as the input to the booking and attendance equations.
\end{examplebox}

\section{Performance Measures}

\subsection{Arrival-Cohort Measures}

For patients who arrive during the measured window, define:
\begin{center}
\small
\begin{tabular}{L{0.26\textwidth} L{0.65\textwidth}}
\toprule

Two utilization measures are reported:
\[
\rho^{\mathrm{booked}}
=
\frac{\mathrm{BookedSlots}}{HS_{\mathrm{measured}}},
\qquad
\rho^{\mathrm{attended}}
=
\frac{\mathrm{ServedSlots}}{HS_{\mathrm{measured}}}.
\]

The first measures scheduled load.
The second measures realized attended load.

\noindent Operationally, mean booking delay summarizes access, \(\rho^{\mathrm{booked}}\)
summarizes scheduled load, and \(\rho^{\mathrm{attended}}\) summarizes realized clinical load after
no-shows.

\section{Relation to the Literature}

\subsection{Literature Mapping}

The source papers use more stylized queue states than the simulator, but the economic and
behavioral interpretation is the same.
In each paper, attendance depends on the congestion level a patient sees when entering the
system.

In the analytical papers, that congestion variable is usually:
\begin{itemize}[nosep]
    \item queue length,
    \item backlog level,
    \item or the number of appointments already ahead in the system.
\end{itemize}

In the simulator, the explicit-calendar analog is the offered delay $\tau$.
That is why $\tau$, rather than residual delay $r$, should drive
$b_i(\cdot)$ and $\xi_i(\cdot)$.

\subsubsection{Green and Savin (2008)}

Green and Savin study a single-server appointment queue in which no-show probability increases
with the backlog seen when the appointment is made.
Their no-show curve is of the exponential-saturation form
\[
\gamma(k) = \gamma_{\max} - (\gamma_{\max} - \gamma_0)e^{-k/C},
\]
where $k$ is a backlog measure.

The mapping to the simulator is:
\begin{itemize}[nosep]
    \item their backlog state corresponds to our explicit slot calendar,
    \item their delay-sensitive no-show mechanism corresponds to our $\xi_i(\tau)$,
    \item their stylized queue-length state is refined into the explicit schedule $Y_t(r,m)$.
\end{itemize}

The simulation differs from Green and Savin in several ways:
\begin{itemize}[nosep]
    \item it allows two patient classes rather than one homogeneous stream,
    \item it includes explicit balking before booking,
    \item it includes daily pre-appointment cancellation that creates explicit holes in the calendar,
    \item it does not yet model no-show rescheduling.
\end{itemize}

\subsubsection{Liu and Ziya (2014)}

Liu and Ziya study panel size and overbooking decisions in stylized queueing models with
delay-dependent attendance.
Their state variable is again a queue-length-type object, and a patient's show-up probability
depends on the congestion level observed at booking time.

Relative to Liu and Ziya, the simulator:
\begin{itemize}[nosep]
    \item fixes daily slot capacity at $25$ in the first version,
    \item does not yet optimize overbooking or panel size,
    \item keeps the booking calendar explicit rather than compressing it into a single queue-length state,
    \item is designed to allow class-specific balking, cancellation, and attendance curves.
\end{itemize}

\subsubsection{Liu (2015)}

Liu studies the optimal appointment-window size $K$ under delay-dependent no-shows.
That paper's main control is the admissible backlog or booking window.

The simulator's rolling horizon length $H$ plays the same operational role as a hard scheduling
window, but here the horizon is fixed in the baseline implementation.
This makes the simulator a natural platform for later experiments in which:
\begin{itemize}[nosep]
    \item $H$ is varied,
    \item different allocation policies are compared,
    \item or class-specific admission rules are introduced.
\end{itemize}

\section{Implementation Notes and Current Limits}

\subsection{Implementation Mapping}

The current codebase follows this notation directly:
\begin{itemize}[nosep]
    \item the slot calendar stores each patient's class, appointment location, booking day, and the mathematical delay $\tau$,
    \item the arrival log records the offered delay as \texttt{tau\_offered},
    \item the cohort and slot logs record the same mathematical quantity $\tau$ under the implementation field name \texttt{tau\_booked},
    \item the day-start state log records the residual day offset as \texttt{residual\_delay},
    \item the class configuration field \texttt{cancel\_probability} stores the class-level eventual cancellation parameter $\bar{\phi}_i$, from which the daily function $\phi_i(\tau)$ is derived in code.
\end{itemize}

This is the same split as in the mathematical formulation.
In addition, the simulator now exposes a daily journal with access, booking, cancellation,
attendance, and utilization measures, and reports the access metric
\[
\mathbb{P}\!\left(\tau < T_{\mathrm{access}}\right)
\]
for a configurable target such as $T_{\mathrm{access}} = 30$ days.

\subsection{Current Scope and Limits}

FCFS is the deliberate first baseline because it makes the delay-feedback mechanism easy to see:
\[
\text{more demand} \longrightarrow \text{longer offered delay } \tau
\longrightarrow \text{more balking / cancellation / no-show}.
\]
That makes it a good diagnostic starting point, even though it should not be interpreted as the
final operational recommendation when classes differ in delay sensitivity.

The implementation is also intentionally a little richer than the minimal mathematical state. The
calendar keeps additional audit information so logs and journals can be produced, but the key
behavioral content still remains the pair \((i,\tau)\). The main provisional simplification is
cancellation: mathematically the note works with \(\phi_i(\tau)\), while in code the current class
configuration still starts from a scalar parameter \(\bar{\phi}_i\) and derives the daily
cancellation function internally. That is acceptable for the first prototype, but the natural next
step is to let the code accept a cancellation function directly if future data support that level
of estimation.
