\section{Purpose}

This note defines the first discrete-event appointment simulation used in this project and
clarifies the notation so it matches the underlying dynamics of the problem.
The main design goal is to keep the first simulator simple enough to run, inspect, and extend,
while using a formulation that remains valid once we add richer allocation policies,
heterogeneous patient behavior, or optimization layers.

The note aligns the simulator with the queueing models in:
\begin{itemize}[nosep]
    \item Green and Savin (2008), \emph{Reducing Delays for Medical Appointments: A Queueing Approach},
    \item Liu and Ziya (2014), \emph{Panel Size and Overbooking Decisions for Appointment-based Services under Patient No-shows},
    \item Liu (2015), \emph{Optimal Choice for Appointment Scheduling Window under Patient No-Show Behavior}.
\end{itemize}

All three papers study the same structural mechanism:
\begin{quote}
more backlog $\Rightarrow$ more delay $\Rightarrow$ worse attendance $\Rightarrow$ lower effective capacity.
\end{quote}

The simulation keeps that mechanism, but implements it on an explicit rolling calendar with
two patient classes, explicit cancellations, and a pluggable slot-allocation rule.

\section{Why the Naive Formulation is Not Sufficient}

An initial formulation might write
\[
X_{i,\tau}^{D} = \text{number of class-$i$ patients booked for day } D+\tau
\]
and then also use $\tau$ inside $b_i(\tau)$ and $\xi_i(\tau)$.
That notation is not the best formulation for this problem.

\subsection{Problem 1: It Overloads the Delay Variable}

There are two different delays in the system:
\begin{enumerate}[nosep]
    \item the \emph{remaining delay} from the current day to the appointment,
    \item the \emph{original delay seen at booking time}.
\end{enumerate}

These play different roles.
The remaining delay is a state variable.
The original delay at booking is the natural argument of the behavioral functions.

If the same symbol $\tau$ is used for both, then the model becomes ambiguous.
In particular, it can incorrectly suggest that no-show probability should be recomputed from
remaining delay as the appointment approaches.
That is not the intended behavioral mechanism here.

\subsection{Problem 2: The Aggregate Day-Level Count is Not the True State}

The vector $\{X_{i,\tau}^{D}\}$ is a useful summary, but it is not the full simulator state once
we include:
\begin{itemize}[nosep]
    \item slot-level capacity,
    \item day-by-day cancellations that reopen specific future slots,
    \item alternative booking policies that may choose different open slots within the same day.
\end{itemize}

For those features, the actual state must retain the slot-level calendar.

\subsection{Problem 3: It Hides the Link to the Literature}

In the source papers, attendance depends on the backlog or delay the patient sees when the
appointment is accepted.
The relevant quantity is therefore the \emph{booking-time delay}, not the residual delay just
before service.

The simulator should preserve that same logic in explicit-calendar form.

\section{Recommended Formulation}

\subsection{Decision Epochs}

Let time advance in slot-sized decision epochs.
Write
\[
t = (D,s),
\]
where:
\begin{itemize}[nosep]
    \item $D$ is the calendar day,
    \item $s \in \{0,1,\dots,S-1\}$ is the active slot within day $D$.
\end{itemize}

In the first version of the simulator,
\[
H = 15 \text{ days}, \qquad S = 25 \text{ slots per day},
\]
where $H$ is the booking horizon.

\subsection{How Time Advances}

The ordered pair $t=(D,s)$ advances lexicographically:
\[
(D,s)^+
=
\begin{cases}
(D,s+1), & \text{if } s < S-1,\\
(D+1,0), & \text{if } s = S-1.
\end{cases}
\]

So the slot index $s$ moves within a day, and the day index $D$ increases only when the
current day has exhausted all $S$ slots.

The same evolution can be visualized as a state-transition graph:

\subsection{True State: Slot-Level Calendar}

The true simulator state at time $t=(D,s)$ is the rolling slot-level calendar.
The mathematical state is indexed by residual day $r$ and slot index $m$.
For visualization, however, it is often more natural to display the calendar with days as
\emph{columns} and slots as \emph{rows}. In that display, the column index corresponds to the day
ahead relative to the current day $D$: $r=0$ means ``today'', $r=1$ means ``tomorrow'', and so on.
The row index corresponds to the slot within a day: $m=0$ is the first slot, $m=1$ the second, and
so on until $m=S-1$.

Formally, for each residual day offset
\[
r \in \{0,1,\dots,H-1\}
\]
and each slot index
\[
m \in \{0,1,\dots,S-1\},
\]
define
\[
Y_t(r,m).
\]

So $Y_t(r,m)$ is the content of the cell indexed by day-ahead $r$ and slot $m$ in the rolling
calendar.
If
\[
Y_t(r,m)=0,
\]
then that slot is open.
If
\[
Y_t(r,m)=\left(i,\tau\right),
\]
then that slot is occupied by a class-$i$ patient who originally accepted delay $\tau$.

The class index $i$ links that booked patient to the class object, written later as
\[
C_i = \{\lambda_i, b_i(\tau), \phi_i(\tau), \xi_i(\tau)\}.
\]
So the cell itself stores only the patient features that must remain attached to the appointment:
the class identity $i$ and the original offered delay $\tau$.
The remaining class characteristics are recovered from $C_i$ when the simulator needs them:
\begin{itemize}[nosep]
    \item $\lambda_i$ governs future arrivals from class $i$,
    \item $b_i(\tau)$ governs booking acceptance when a class-$i$ request is offered delay $\tau$,
    \item $\phi_i(\tau)$ governs daily pre-appointment cancellation for class-$i$ patients booked with delay $\tau$,
    \item $\xi_i(\tau)$ governs no-show at appointment time.
\end{itemize}

Schematically, the rolling calendar has the following structure:

This example is read column by column.
The first column is today's remaining schedule, the second column is tomorrow's schedule, and the
third column is the schedule for the day after tomorrow.
For instance, the entry in day-column $r=1$ and slot-row $m=2$ is $(2,1)$, meaning that the
13:00 slot of day $D+1$ is occupied by a class-2 behavioral-health patient who accepted a
one-day delay.
Because the patient belongs to class $2$, the simulator knows to use the class-2 behavior object
$C_2=\{\lambda_2,b_2(\tau),\phi_2(\tau),\xi_2(\tau)\}$.
That means:
\begin{itemize}[nosep]
    \item future class-2 arrivals are generated with rate $\lambda_2$,
    \item the patient's daily pre-appointment cancellation behavior is governed by $\phi_2(1)$,
    \item if the patient reaches service, the no-show probability is $\xi_2(1)$ because the original booked delay is $\tau=1$.
\end{itemize}

This is the minimal state representation that supports:
\begin{itemize}[nosep]
    \item explicit holes created by daily cancellation,
    \item slot-level FCFS or non-FCFS allocation,
    \item class-specific booked-delay tracking,
    \item later policy experiments that depend on slot position.
\end{itemize}

\subsection{Derived Day-Start Summary}

Although the slot-level calendar is the true state, it is still useful to retain an aggregate
day-start summary that resembles the literature.
At the start of day $D$, after the previous day's end-of-day cancellations and before processing
slot $0$,
define
\[
X_{i,r}^{D}
=
\sum_{m=0}^{S-1}
\mathbf{1}\{Y_{(D,0)}(r,m)\text{ is occupied by a class-}i\text{ patient}\},
\]
for
\[
i \in \{1,2\}, \qquad r \in \{0,1,\dots,H-1\}.
\]

In words, $X_{i,r}^{D}$ counts how many class-$i$ patients appear in row $r$ of the calendar at
the start of day $D$.
Equivalently,
\[
X_{i,r}^{D}
=
\text{number of class-}i\text{ patients scheduled at the start of day }D\text{ for day }D+r.
\]

This is the correct interpretation of the state-count variable.
The index $r$ is a \emph{residual delay}, not a booking-time delay.

\subsection{Worked Example of Y and X}

It is useful to see one concrete calendar snapshot and translate it directly into the state and
equations.
Suppose:
\[
D=0, \qquad s=0, \qquad H=3, \qquad S=4.
\]
So we are at the start of the current operating day, just before the 08:00 slot, and the rolling
calendar covers today, tomorrow, and the day after tomorrow.

Assume the slot-level state is:
\[
\begin{array}{c|cccc}
& \text{08:00} & \text{10:00} & \text{13:00} & \text{15:00} \\
\hline
r=0\text{ (today)} & 0 & (1,0) & (2,0) & 0 \\
r=1\text{ (tomorrow)} & (1,1) & 0 & (1,1) & (2,1) \\
r=2\text{ (day after)} & 0 & (2,2) & (1,2) & 0
\end{array}
\]

Interpretation:
\begin{itemize}[nosep]
    \item row $r=0$ corresponds to today's remaining slots,
    \item row $r=1$ corresponds to tomorrow,
    \item row $r=2$ corresponds to the day after tomorrow,
    \item entry $0$ means the slot is open,
    \item entry $(i,\tau)$ means the slot is occupied by a class-$i$ patient who accepted an offered delay $\tau$.
\end{itemize}

For example:
\begin{itemize}[nosep]
    \item $Y_{(0,0)}(0,1)=(1,0)$ means that today's 10:00 slot is booked by a class-1 MRI patient who accepted a same-day offer,
    \item $Y_{(0,0)}(1,0)=(1,1)$ means that tomorrow's 08:00 slot is booked by a class-1 MRI patient who accepted a one-day delay,
    \item $Y_{(0,0)}(2,1)=(2,2)$ means that the 10:00 slot two days from now is booked by a class-2 behavioral-health patient who accepted a two-day delay.
\end{itemize}

For the second patient, the state stores only $(2,2)$.
That is enough because class index $2$ points to
\[
C_2=\{\lambda_2,b_2(\tau),\phi_2(\tau),\xi_2(\tau)\}.
\]
So while the patient waits, the simulator uses the class-2 cancellation function $\phi_2(\tau)$ to
apply the daily cancellation probability $\phi_2(2)$, and if the patient reaches service it uses $\xi_2(2)$ for
attendance.

From this calendar, the derived day-start counts are:
\[
X_{1,0}^{0}=1,\quad X_{2,0}^{0}=1,
\]
\[
X_{1,1}^{0}=2,\quad X_{2,1}^{0}=1,
\]
\[
X_{1,2}^{0}=1,\quad X_{2,2}^{0}=1.
\]

So the vector $X_{i,r}^{0}$ is just a class-by-class count of the occupied entries in each row.
It does \emph{not} remember which slot inside the day is occupied, nor does it retain the
full slot-level identity of individual patients.
That is why $X_{i,r}^{D}$ is a useful summary, but not the full state needed by the simulator.

The distinction between $r$ and $\tau$ is also visible in this example.
Consider the patient stored as
\[
Y_{(0,0)}(2,2)=(1,2).
\]
On day $0$, that patient contributes to $X_{1,2}^{0}$.
If the patient remains scheduled, then at the start of day $1$ the same patient will contribute
to $X_{1,1}^{1}$, because the \emph{residual delay} has fallen from $2$ to $1$.
However, the patient's booking-time delay remains $\tau=2$, and that is the quantity that should
continue to drive the behavioral functions $b_1(\tau)$ and $\xi_1(\tau)$.

\subsection{From Calendar Story to Equations}

The realistic booking story above transfers directly into the mathematical model.
Suppose a new class-2 patient arrives just before today's 08:00 slot and the FCFS rule offers the
open slot at 08:00 two days from now.
Then:
\begin{itemize}[nosep]
    \item the offered delay is $\tau=2$ because the offered day is $D+2$,
    \item the patient books with probability $1-b_2(2)$,
    \item if the patient accepts, the new slot content becomes
    \[
    Y_{(0,0)}(2,0)=(2,2),
    \]
    \item at the end of each pre-appointment day, that patient cancels with daily probability $\phi_2(2)$,
    \item if the patient reaches the appointment, attendance occurs with probability $1-\xi_2(2)$.
\end{itemize}

So the same appointment can be read in three equivalent ways:
\begin{itemize}[nosep]
    \item as a realistic story: ``a behavioral-health follow-up booked for two days from now,''
    \item as a state entry: $Y_t(r,m)=(2,2)$ in the appropriate cell,
    \item as behavioral equations: booking through $b_2(2)$, cancellation through $\phi_2(2)$, and attendance through $\xi_2(2)$.
\end{itemize}

\subsection{Booking-Time Delay}

Now define the behavioral delay variable separately:
\[
\tau
=
\text{delay in days between the booking day and the appointment day offered to the patient.}
\]

If a patient arrives on day $d$ and is offered a slot on day $d+\tau$, then:
\begin{itemize}[nosep]
    \item $b_i(\tau)$ is the probability that class-$i$ declines the offer,
    \item if the patient accepts, $\tau$ is stored with the booking,
    \item $\xi_i(\tau)$ is the no-show probability used at appointment time if the patient remains scheduled.
\end{itemize}

This separation is the key modeling choice.
Residual delay belongs to the state.
Booking delay belongs to the behavioral rules.

\section{Patient Classes and Arrival Mix}

The class-specific behavior of each class $i \in \{1,2\}$ is described by
\[
B_i = \{b_i(\tau), \phi_i(\tau), \xi_i(\tau)\}.
\]

In the current two-class prototype, arrivals are parameterized at the system level by
\[
\lambda
=
\text{total expected arrival rate per slot},
\qquad
p \in [0,1]
=
\text{share of class-1 demand}.
\]

The class-specific arrival rates are then induced by
\[
\lambda_1 = p\lambda,
\qquad
\lambda_2 = (1-p)\lambda.
\]

Equivalently, the full class objects can still be written as
\[
C_i = \{\lambda_i, b_i(\tau), \phi_i(\tau), \xi_i(\tau)\},
\]
but in the first simulation experiments the primitive demand parameters are
$(\lambda,p)$ rather than $(\lambda_1,\lambda_2)$.

The interpretation is therefore:
\begin{itemize}[nosep]
    \item $\lambda$: total expected arrival rate per slot,
    \item $p$: fraction of arrivals belonging to class 1,
    \item $\lambda_i$: implied arrival rate of class-$i$ requests per slot,
    \item $b_i(\tau)$: probability the patient declines the offered slot because the offered delay is too large,
    \item $\phi_i(\tau)$: daily probability that a class-$i$ patient booked with original delay $\tau$ cancels at the end of a pre-appointment day,
    \item $\xi_i(\tau)$: no-show probability at appointment time, conditional on the patient still being scheduled.
\end{itemize}

In the current first version, the daily cancellation function $\phi_i(\tau)$ is induced by a
class-level eventual cancellation parameter $\bar{\phi}_i$ according to
\[
\phi_i(\tau)=
\begin{cases}
0, & \tau=0,\\[0.4em]
1-(1-\bar{\phi}_i)^{1/\tau}, & \tau\ge 1,
\end{cases}
\]
so that a patient who booked with delay $\tau$ and faces one cancellation trial at the end of each
pre-appointment day still has total cancellation probability $\bar{\phi}_i$ before the appointment.
Later versions may generalize this further to a directly delay-sensitive cancellation object.

\section{Event Dynamics}

\subsection{Slot-Level Event Order}

At each active slot $(D,s)$, the simulator executes the following within-day steps.

\begin{enumerate}[leftmargin=*, itemsep=0.35em]
    \item Generate arrivals
    \[
    A_i(D,s) \sim \mathrm{Poisson}(\lambda_i), \qquad i \in \{1,2\}.
    \]
    In the current parameterization, $\lambda_1=p\lambda$ and $\lambda_2=(1-p)\lambda$.
    \item Randomize the order of individual arrivals within the slot.
    \item For each arrival, call the allocation policy to select an open future slot.
    \item If a slot on day $D+r$ is offered, interpret
    \[
    \tau = r.
    \]
    \item The patient accepts with probability
    \[
    1 - b_i(\tau).
    \]
    \item When the active slot is served, the patient in that slot, if any, attends with probability
    \[
    1 - \xi_i(\tau),
    \]
    where $\tau$ is the original delay stored when the patient booked.
\end{enumerate}

After all $S$ slots of day $D$ have been processed, every still-scheduled future appointment
stored as $(i,\tau)$ cancels independently with probability $\phi_i(\tau)$.
Any canceled future slot is therefore reopened before day $D+1$ begins.
This timing is operationally natural:
\begin{itemize}[nosep]
    \item a booking with $\tau=1$ gets exactly one chance to cancel, namely at the end of the booking day,
    \item a booking with $\tau=2$ gets two daily cancellation opportunities before service,
    \item the total cancellation probability across those daily trials remains $\bar{\phi}_i$.
\end{itemize}

\subsection{FCFS Baseline Policy}

In the baseline version, the allocation rule is:
\begin{quote}
choose the earliest open slot strictly after the active slot.
\end{quote}

That rule is first-come first-served in calendar order, but the simulator is intentionally
written so the slot-selection rule can later be replaced without changing the event loop.
The current codebase now includes two additional initial-model examples:
\begin{itemize}[nosep]
    \item a strict reserved-capacity policy that protects a specified fraction or count of daily slots for each class,
    \item a class-specific booking-window policy that restricts class $i$ to delays no larger than $k_i$.
\end{itemize}

\section{Performance Measures}

\subsection{Arrival-Cohort Measures}

For patients who arrive during the measured window, define:
\begin{align*}
\mathrm{Arrivals}_i &= \text{number of class-}i\text{ appointment requests}, \\
\mathrm{Booked}_i &= \text{number of class-}i\text{ requests that accept an offer}, \\
\mathrm{Balked}_i &= \text{number of class-}i\text{ requests that were offered a slot and declined it}, \\
\mathrm{NoOffer}_i &= \text{number of class-}i\text{ requests for which no eligible slot was available}, \\
\mathrm{NotBooked}_i &= \mathrm{Balked}_i + \mathrm{NoOffer}_i, \\
\mathrm{Canceled}_i &= \text{number of class-}i\text{ booked patients who pre-cancel}, \\
\mathrm{NoShow}_i &= \text{number of class-}i\text{ booked patients who remain scheduled but do not attend}, \\
\mathrm{Served}_i &= \text{number of class-}i\text{ booked patients who attend}, \\
\bar{\tau}^{\mathrm{book}}_i &= \text{mean original booking delay among class-}i\text{ booked patients}.
\end{align*}

The main service metric in the first version is
\[
\mathrm{ServiceRate}_i
=
\frac{\mathrm{Served}_i}{\mathrm{Booked}_i}.
\]

Aggregate values are formed by summing over classes and taking the corresponding weighted mean.

\subsection{Slot-Based Measures}

Over measured service slots, define:
\begin{align*}
\mathrm{BookedSlots} &= \text{slots with a booked patient at service time}, \\
\mathrm{ServedSlots} &= \text{slots where the booked patient attends}, \\
\mathrm{NoShowSlots} &= \text{slots where a booked patient is present on the schedule but does not attend}, \\
\mathrm{EmptySlots} &= \text{slots with nobody scheduled at service time}.
\end{align*}

Two utilization measures are reported:
\[
\rho^{\mathrm{booked}}
=
\frac{\mathrm{BookedSlots}}{HS_{\mathrm{measured}}},
\qquad
\rho^{\mathrm{attended}}
=
\frac{\mathrm{ServedSlots}}{HS_{\mathrm{measured}}}.
\]

The first measures scheduled load.
The second measures realized attended load.

\section{Comparison to the Literature}

\subsection{Common Structural Mapping}

The source papers use more stylized queue states than the simulator, but the economic and
behavioral interpretation is the same.
In each paper, attendance depends on the congestion level a patient sees when entering the
system.

In the analytical papers, that congestion variable is usually:
\begin{itemize}[nosep]
    \item queue length,
    \item backlog level,
    \item or the number of appointments already ahead in the system.
\end{itemize}

In the simulator, the explicit-calendar analog is the offered delay $\tau$.
That is why $\tau$, rather than residual delay $r$, should drive
$b_i(\cdot)$ and $\xi_i(\cdot)$.

\subsection{Green and Savin (2008)}

Green and Savin study a single-server appointment queue in which no-show probability increases
with the backlog seen when the appointment is made.
Their no-show curve is of the exponential-saturation form
\[
\gamma(k) = \gamma_{\max} - (\gamma_{\max} - \gamma_0)e^{-k/C},
\]
where $k$ is a backlog measure.

The mapping to the simulator is:
\begin{itemize}[nosep]
    \item their backlog state corresponds to our explicit slot calendar,
    \item their delay-sensitive no-show mechanism corresponds to our $\xi_i(\tau)$,
    \item their stylized queue-length state is refined into the explicit schedule $Y_t(r,m)$.
\end{itemize}

The simulation differs from Green and Savin in several ways:
\begin{itemize}[nosep]
    \item it allows two patient classes rather than one homogeneous stream,
    \item it includes explicit balking before booking,
    \item it includes daily pre-appointment cancellation that creates explicit holes in the calendar,
    \item it does not yet model no-show rescheduling.
\end{itemize}

The most important conceptual similarity is this:
attendance deteriorates with the delay the patient faced when entering the schedule.

\subsection{Liu and Ziya (2014)}

Liu and Ziya study panel size and overbooking decisions in stylized queueing models with
delay-dependent attendance.
Their state variable is again a queue-length-type object, and a patient's show-up probability
depends on the congestion level observed at booking time.

Relative to Liu and Ziya, the simulator:
\begin{itemize}[nosep]
    \item fixes daily slot capacity at $25$ in the first version,
    \item does not yet optimize overbooking or panel size,
    \item keeps the booking calendar explicit rather than compressing it into a single queue-length state,
    \item is designed to allow class-specific balking, cancellation, and attendance curves.
\end{itemize}

Their model is therefore more compact and optimization-oriented.
Our simulator is more descriptive and state-rich.
The key shared mechanism remains the same:
\[
\text{booking-time congestion} \longrightarrow \text{attendance}.
\]

\subsection{Liu (2015)}

Liu studies the optimal appointment-window size $K$ under delay-dependent no-shows.
That paper's main control is the admissible backlog or booking window.

The simulator's rolling horizon length $H$ plays the same operational role as a hard scheduling
window, but here the horizon is fixed in the baseline implementation.
This makes the simulator a natural platform for later experiments in which:
\begin{itemize}[nosep]
    \item $H$ is varied,
    \item different allocation policies are compared,
    \item or class-specific admission rules are introduced.
\end{itemize}

Again, the important mapping is not the exact state variable but the interpretation:
the patient's future attendance depends on the delay accepted at booking time.

\section{Why this Formulation is the Most Appropriate for This Problem}

This formulation is the most appropriate starting point for the present project for four reasons.

\subsection{1. It Matches the Behavioral Mechanism}

The patient decides whether to book when seeing an offered delay.
If they book, their future no-show behavior should be tied to that accepted delay.
That is exactly what $\tau$ captures.

\subsection{2. It Preserves the Literature's Logic While Using an Explicit Calendar}

The source papers compress the calendar into queue length for tractability.
The simulator does not need to do that.
It can retain the slot-level calendar and still summarize the system in a literature-friendly way
through $X_{i,r}^{D}$.

\subsection{3. It Is Markov for the Planned Extensions}

Once cancellations create holes and policies can choose among open slots, a pure day-level count
is no longer enough.
The calendar state $Y_t(r,m)$ is the right object to carry forward if the project later adds:
\begin{itemize}[nosep]
    \item priority rules,
    \item reserved capacity by class,
    \item different appointment windows by class,
    \item overbooking,
    \item dynamic acceptance or rejection decisions.
\end{itemize}

\subsection{4. It Cleanly Separates State and Behavior}

The formulation distinguishes:
\begin{itemize}[nosep]
    \item \emph{where the system is now}: represented by residual delay $r$ in the state,
    \item \emph{what delay the patient accepted}: represented by $\tau$ in the behavior functions.
\end{itemize}

That separation is the cleanest way to avoid conceptual mistakes in both code and analysis.

\section{Implementation Mapping}

The current codebase follows this notation directly:
\begin{itemize}[nosep]
    \item the slot calendar stores each patient's class, appointment location, booking day, and the mathematical delay $\tau$,
    \item the arrival log records the offered delay as \texttt{tau\_offered},
    \item the cohort and slot logs record the same mathematical quantity $\tau$ under the implementation field name \texttt{tau\_booked},
    \item the day-start state log records the residual day offset as \texttt{residual\_delay},
    \item the class configuration field \texttt{cancel\_probability} stores the class-level eventual cancellation parameter $\bar{\phi}_i$, from which the daily function $\phi_i(\tau)$ is derived in code.
\end{itemize}

This is exactly the split recommended above.
In addition, the simulator now exposes a daily journal with access, booking, cancellation,
attendance, and utilization measures, and reports the access metric
\[
\mathbb{P}\!\left(\tau < T_{\mathrm{access}}\right)
\]
for a configurable target such as $T_{\mathrm{access}} = 30$ days.

\section{Strategy and Code Review}

\subsection{Why FCFS is the Right First Strategy}

FCFS is the right baseline because it is transparent, easy to explain to operational partners, and
reveals the core mechanism of the project with minimal confounding:
\[
\text{more demand} \longrightarrow \text{longer offered delay } \tau
\longrightarrow \text{more balking / cancellation / no-show}.
\]

That makes FCFS an excellent diagnostic benchmark.
However, it should not be treated as the final strategic recommendation, because it ignores class
heterogeneity.
If classes differ materially in access sensitivity or attendance sensitivity, FCFS may be simple
but not efficient.

\subsection{Why the Code Structure is Sound}

The implementation is well aligned with the mathematical formulation for three reasons:
\begin{itemize}[nosep]
    \item the policy rule is isolated behind an allocation interface, so FCFS can later be replaced without rewriting the event loop,
    \item the implementation state is richer than the mathematical state, but only for auditing and measurement; the minimal behavioral content is still the pair $(i,\tau)$,
    \item the simulator exposes arrival logs, cohort logs, slot logs, state logs, and daily journals, so every key equation in the note can be checked against an observable output table.
\end{itemize}

\subsection{Current Strategic and Coding Limits}

The main provisional simplification is cancellation.
Mathematically, the note now works with the daily function $\phi_i(\tau)$.
In code, the current class configuration still starts from a scalar parameter $\bar{\phi}_i$ and
derives $\phi_i(\tau)$ internally.
That is acceptable for the first prototype because it keeps calibration simple, but if future data
support direct estimation of delay-specific cancellation behavior, the natural next step is to let
the code accept a cancellation function directly, just as it already does for balking and no-show.
